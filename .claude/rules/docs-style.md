# Docs Style

All technical documentation must reflect the actual behavior of the code — not future plans or assumptions.

## Two folders, two purposes

The base reserves two distinct folders for documentation:

| Folder | Purpose | Audience |
|---|---|---|
| `documentation/` | **Code documentation** — generated by `/document`, maintained by the team | Developers, future maintainers |
| `docs/` | **GitHub Pages landing site** — public-facing project page | External users, recruiters, demo visitors |

These do **not overlap**. `/document` and `doc-enforce` always target `documentation/`.
GitHub Pages always publishes from `docs/`.

If you're tempted to put code docs in `docs/`, stop — they go to `documentation/`.
If you're tempted to put landing-page assets in `documentation/`, stop — they go to `docs/`.

## Filename convention (for `documentation/`)

- See `file-naming.md` for naming guidance.
- Pattern: `NN_<slug>.md` for ordered docs (e.g., `01_architecture.md`, `02_loader.md`).
- Files live at the root of `documentation/` unless the project has many docs (then organize into subfolders).

## Required sections (for code docs in `documentation/`)

Every code documentation file must include:

1. **Title and Purpose** — Start with a single sentence summarizing the module's role.
2. **Workflow Description** — Sequence of operations performed. Numbered steps or descriptive text. Include a Mermaid diagram (`flowchart TD` or `graph LR`) if the flow is complex.
3. **Inputs and Outputs** — List each parameter with name, type, and purpose. Describe expected output(s) and their structure.
4. **Mathematical Explanation** (if applicable) — Use LaTeX or pseudocode for formulas.
5. **Code Reference** — Always include the source module path (e.g., `Source: src/module.py`).

## Style

- Write clearly and concisely. Short paragraphs and examples when helpful.
- Document only what exists in the current version of the code.
- Avoid TODOs and speculative notes.
- Each document must be self-contained — readable without prior context.

## Bilingual docs

If the project uses bilingual code docs, organize by language subfolder under `documentation/`:

```
documentation/
├── English/
│   └── 01_architecture.md
└── Spanish/
    └── 01_arquitectura.md
```

GitHub Pages bilingual sites have a different convention (Jekyll multilang plugin, language switcher in `docs/`).

## GitHub Pages site (`docs/`)

The base does not prescribe how to build the landing page. Common stacks:

- Plain HTML/CSS/JS at `docs/index.html`
- Jekyll (default for GitHub Pages) — `docs/_config.yml` + Markdown
- A static site generator's build output (Astro, Hugo, MkDocs)

`docs/` should be configured in GitHub Pages settings as the source folder.
The base does not enforce a structure inside it — that depends on the chosen stack.
